# 人才打标系统 - 应用架构描述

## 1. 系统概述

人才打标系统是一套以标签资产为核心、以规则治理为驱动、以 AI 能力贯穿全链路的人才标签治理平台。系统支持结构化规则打标和 AI 语义规则打标两种模式，通过统一的标签资产管理、规则生命周期治理和结果证据追溯，实现人才标签的标准化、可解释、可治理。

## 4. 后端架构

### 4.1 分层结构

```
com.talent.label/
├── controller/     ← REST API 入口（11 个控制器）
├── service/        ← 业务逻辑层（含 impl 实现）
├── domain/
│   ├── entity/     ← 数据实体（22 个）
│   ├── dto/        ← 数据传输对象
│   └── enums/      ← 枚举定义
├── mapper/         ← MyBatis-Plus 数据访问层（20+ Mapper）
├── ai/
│   ├── tools/      ← AI 工具实现（12 个 FunctionCallback）
│   ├── DynamicChatModelFactory  ← 动态模型工厂
│   └── AssistantPrompts         ← 系统提示词构建
├── config/         ← 配置类（CORS、MyBatis、AI 工具注册）
├── scheduler/      ← 定时任务（变更感知）

```

| TagRuleController | /tag-rules | 规则 CRUD、发布、撤销、复制 |

### 4.3 核心业务服务

| 服务 | 职责 |
|------|------|
| CalcTaskService | 任务生命周期管理（创建、运行、提交、撤销、删除） |
| CalcTaskExecutor | 异步打标执行引擎（结构化规则 + AI 语义规则） |
| DslParser | JSON DSL 解析与条件评估 |
| TagRuleService | 规则生命周期（编辑、发布、撤销、复制、删除） |
| AiService | AI 能力封装（DSL 生成、语义打标、方案规划） |
| ChatAssistantService | AI 对话管理（SSE 流式、工具调用、操作确认） |

### 4.4 打标执行引擎

CalcTaskExecutor 是系统核心，负责异步执行打标任务：

```
任务运行
  → 获取关联规则列表
  → 确定目标员工范围（全员 / 按组织 / 按员工）
  → 清理旧结果（支持重新运行）
  → 逐规则执行：
      ├── 结构化规则：DslParser 解析 JSON → 逐员工评估条件 → 写入证据
      └── AI 语义规则：构建输入文本 → 调用 LLM → 解析返回 → 写入证据
  → 更新任务状态（SUCCESS / FAILED）
```

## 6. AI 能力架构

### 6.1 模型管理

系统通过 DynamicChatModelFactory 支持多模型动态切换：

- 支持 OPENAI、AZURE、DEEPSEEK 三种提供商
- 数据库配置模型参数（temperature、max_tokens）
- 运行时缓存模型实例，按需创建
- 默认使用 GPT-4o

### 6.2 AI 工具体系（12 个技能）

```
┌─────────────────────────────────────────┐
│              AI 工具层                    │
├─────────────┬───────────────────────────┤
│ 对象操作(6)  │ manage_category           │
│             │ manage_tag                │
│             │ manage_rule               │
│             │ manage_task               │
│             │ manage_approval           │
│             │ query_employee            │
├─────────────┼───────────────────────────┤
│ 关联分析(4)  │ check_references          │
│             │ analyze_impact            │
│             │ analyze_change            │
│             │ generate_dsl              │
├─────────────┼───────────────────────────┤
│ 系统辅助(2)  │ confirm_operation         │
│             │ get_tag_stats             │
└─────────────┴───────────────────────────┘
```

- 写操作（MUTATION）需用户确认后执行
- 查询操作（QUERY）直接返回结果
- 工具通过 Spring AI FunctionCallback 注册
- 数据库管理技能启用/禁用状态

### 6.3 AI 对话流程

```
用户发送消息
  → 构建 System Prompt（注入技能描述 + 未读通知 + 页面上下文）
  → 加载历史消息（最近 20 条）
  → 调用 ChatClient（SSE 流式）
  → 模型按需调用工具（Function Calling）
  → 工具执行结果回传模型
  → 最终响应流式返回前端
```

## 7. 数据库架构

### 7.1 核心表关系

```
tag_category (标签类目)
  └── 1:N → tag_definition (标签定义)
                └── N:M → tag_rule (打标规则) [通过 DSL 引用]
                              ├── 1:N → calc_task_rule → calc_task (打标任务)
                              └── 1:N → employee_tag_result_detail (证据)
                                          └── → employee_tag_result (标签结果)
                                                    └── → employee (员工)
```

### 7.3 数据库迁移

通过 Flyway 管理 12 个版本的迁移脚本（V1-V12），覆盖：
- V1：核心表结构（16 张表）
- V2-V5：AI 技能系统演进
- V6：员工变更追踪（PostgreSQL 触发器）
- V7-V8：规则状态简化与 DSL 格式更新
- V9：技能层重设计（12 个聚合技能）
- V10-V12：规则标签关联修复与技能描述更新

## 8. 变更感知机制

```
员工数据变更（外部系统写入）
  → PostgreSQL 触发器捕获字段级变更
  → 写入 employee_change_log 表
  → 定时任务（60s）扫描未处理变更
  → 匹配受影响的已发布规则
  → 生成 change_notification（INFO / WARN / CRITICAL）
  → 前端铃铛展示 + AI 对话上下文注入
  → 员工离职自动失效所有标签
```

## 10. 核心业务流程

### 10.1 标签打标全链路

```
定义标签类目 → 定义标签 → 创建规则（引用标签）→ 发布规则
  → 模拟打标（验证）→ 正式打标（执行）→ 提交审批 → 审批通过 → 标签生效
```

### 10.2 规则编辑权限

| 条件 | 可编辑 | 可删除 |
|------|--------|--------|
| 未被正式任务使用 | ✓ | ✓（未被任何任务引用时） |
| 被正式任务运行成功/运行中 | ✗（需先撤销任务） | ✗（被任务引用） |

### 10.3 标签生效链路

```
运行成功（validFlag=false）
  → 提交（SUBMITTED）
  → 审批通过（APPROVED，validFlag=true）
  → 标签总览可见
```
