# 人才打标系统数据库表结构草案

## 1. 设计原则
- 标签和规则统一定义，不按组织分别建模。
- 通过角色权限控制标签和规则的创建、维护、发布。
- 规则由两部分组成：命中条件、适用范围。
- 适用范围支持组织、岗位序列、职位族，并可组合。
- 标签类目区分互斥类目与可并存类目。
- 打标结果需支持可解释、可追溯、可重算。

## 2. 表清单
建议首版包含以下 12 张核心表：
- `employee`
- `tag_category`
- `tag_definition`
- `tag_rule`
- `tag_rule_condition`
- `tag_rule_scope`
- `employee_tag_result`
- `employee_tag_result_detail`
- `calc_task`
- `sys_role`
- `sys_user_role`
- `operation_log`

## 3. 表结构草案

### 3.1 employee

| --- | --- | --- | --- |

| name | varchar(128) | 是 | 姓名 |

| job_family_code | varchar(64) | 否 | 职位族编码 |
| job_family_name | varchar(128) | 否 | 职位族名称 |
| job_title | varchar(128) | 否 | 职称/岗位名称 |
| grade_level | varchar(64) | 否 | 职级，如P7 |
| birth_date | date | 否 | 出生日期，年龄由此计算 |

| employee_status | varchar(32) | 是 | 在职/离职/冻结 |
| updated_at | datetime | 是 | 最近更新时间 |

### 3.5 tag_rule_condition
用途：规则命中条件，支持结构化表达。

| 字段名 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| id | bigint | 是 | 主键 |
| rule_id | bigint | 是 | 规则ID |
| group_no | int | 是 | 条件组编号，首版可默认1 |
| logic_operator | varchar(16) | 是 | 与前一条件关系：AND/OR |
| field_code | varchar(64) | 是 | 字段编码，如grade_level、age、job_title |
| field_name | varchar(128) | 是 | 字段名称 |
| operator | varchar(32) | 是 | EQ/IN/GE/GT/LE/LT/BETWEEN |
| value_type | varchar(32) | 是 | STRING/NUMBER/DATE/ENUM |
| value_expr | varchar(500) | 是 | 条件值表达式 |
| sort_no | int | 是 | 排序号 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 修改时间 |

说明：
- `value_expr` 建议存 JSON 字符串，兼容单值、多值、区间。
- 年龄字段为派生字段，不直接来自员工表持久列。

示例：
- `field_code = grade_level`，`operator = EQ`，`value_expr = "P7"`
- `field_code = age`，`operator = BETWEEN`，`value_expr = {"left":20,"right":30,"leftInclusive":true,"rightInclusive":false}`

建议索引：
- 普通索引：`rule_id`
- 普通索引：`field_code`

### 3.6 tag_rule_scope
用途：规则适用范围，限定规则对哪些人参与判断。

| 字段名 | 类型 | 是否必填 | 说明 |
| --- | --- | --- | --- |
| id | bigint | 是 | 主键 |
| rule_id | bigint | 是 | 规则ID |
| scope_type | varchar(32) | 是 | ORG/POSITION_SEQUENCE/JOB_FAMILY |
| scope_value | varchar(256) | 是 | 范围值，可存编码或ID |
| scope_name | varchar(128) | 是 | 范围名称 |
| include_children_flag | tinyint | 否 | 组织场景是否包含下级 |
| logic_group | int | 是 | 逻辑组编号，首版默认1 |
| sort_no | int | 是 | 排序号 |
| created_at | datetime | 是 | 创建时间 |
| updated_at | datetime | 是 | 修改时间 |

规则建议：
- 同一 `scope_type` 下多条记录按 OR 处理。
- 不同 `scope_type` 之间按 AND 处理。

建议索引：
- 普通索引：`rule_id`
- 普通索引：`scope_type, scope_value`

## 4. 关键约束与业务规则

### 4.1 标签类目互斥规则
- `tag_category.exclusive_flag = 1` 表示互斥类目。
- 同一员工在互斥类目下最终只能保留一个有效标签。
- 冲突时按以下顺序决策：
  1. 条件更精确者优先。
  2. 若无法比较精确度，则按 `tag_rule.priority`。
  3. 若仍冲突，则按发布时间或规则ID兜底，并记录到 `employee_tag_result_detail`。

### 4.2 年龄口径
- 不建议持久化年龄字段。
- 运行时按 `birth_date` 计算年龄，避免数据过期。

### 4.3 范围判断口径
- 组织范围支持“本组织”和“本组织及下级”。
- 岗位序列、职位族范围支持多选。
- 同维度多值 OR，不同维度 AND。

### 4.4 版本口径
- 已发布规则修改时，建议生成新版本，而不是直接覆盖。
- 历史版本保留，便于审计和回溯。

## 5. 首版不建议纳入数据库的能力
以下能力建议二期再考虑，不必在首版表结构中提前复杂化：
- 通用脚本表达式引擎
- 复杂括号逻辑嵌套
- 实时流式计算
- 多级审批流程引擎
- 标签统计分析宽表
