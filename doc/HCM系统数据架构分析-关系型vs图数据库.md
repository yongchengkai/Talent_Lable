# HCM 系统数据架构分析 — 关系型 vs 图数据库

## 1. 背景

当系统从人才打标扩展到完整 HCM（Human Capital Management）时，数据关系的复杂度会显著上升。本文分析在 HCM 场景下，关系型数据库和图数据库各自的适用边界，并给出混合架构建议。

## 2. HCM 关系复杂度分析

### 2.1 当前打标系统的关系模型

打标系统的关系本质上是树状 + 有限层级的链式关系：

```
类目 → 标签 → 规则 → 任务 → 结果 → 员工
```

- 最深 5-6 层，SQL JOIN 完全覆盖
- 关系方向明确，不存在环路
- 查询模式固定，不需要动态路径发现

**结论：关系型数据库足够，不需要图数据库。**

### 2.2 完整 HCM 的关系模型

```
员工 ←→ 组织（矩阵汇报、虚线汇报、兼岗）
员工 ←→ 岗位（职族、职级、职位序列、任职资格）
员工 ←→ 员工（上下级、导师、项目协作、继任关系）
员工 ←→ 能力（技能树、认证路径、培训依赖）
岗位 ←→ 能力（胜任力模型、岗位要求）
组织 ←→ 组织（合并、拆分、历史沿革）
```

关键差异：

| 维度 | 打标系统 | 完整 HCM |
|------|---------|---------|
| 关系层级 | 5-6 层，单向链式 | 10+ 层，多向网状 |
| 关系类型 | 固定（引用、关联） | 动态（汇报、协作、继任、发展） |
| 查询模式 | 固定 JOIN | 多跳遍历、路径发现、相似度匹配 |
| 环路 | 无 | 有（互为导师、交叉汇报） |

## 3. 关系型数据库的能力边界

### 3.1 擅长的场景

- 员工主数据 CRUD（入职、调薪、请假、考勤）
- 薪酬计算、社保公积金等结构化计算
- 审批流程、操作日志
- 标签资产、规则、任务管理
- 所有需要 ACID 事务保证的写操作
- 合规审计（数据一致性要求高）

### 3.2 吃力的场景

| 场景 | SQL 实现方式 | 问题 |
|------|------------|------|
| 继任链路分析 | 递归 CTE | 3 层以上性能急剧下降，代码可读性差 |
| 多维人才寻访 | 5-6 个 JOIN | 查询复杂度高，优化困难 |
| 组织影响分析 | 多层递归 + 关联查询 | 级联影响难以穷举 |
| 能力差距分析 | 交叉 JOIN + 聚合 | 图上的相似度匹配更自然 |
| 职业发展路径推荐 | 几乎无法实现 | 需要路径遍历和权重计算 |
| 动态关系发现 | 不支持 | "这两个人之间有什么关联"无法用固定 SQL 回答 |

### 3.3 性能拐点

```
JOIN 层数    响应时间（万级数据）
  2 层        < 50ms
  3 层        < 200ms
  4 层        500ms - 2s
  5 层        2s - 10s
  6 层+       不可控
```

HCM 的关系分析查询经常需要 4-6 层 JOIN，在数据量增长后会成为瓶颈。

## 4. 图数据库的适用场景

### 4.1 图数据库的核心优势

- **多跳遍历**：3 层以上的关系查询性能稳定，不随深度线性下降
- **路径发现**：找到两个节点之间的所有路径（最短路径、加权路径）
- **模式匹配**：在图中发现符合特定模式的子图
- **相似度计算**：基于图结构的节点相似度（共同邻居、Jaccard 系数）
- **影响传播**：从一个节点出发，计算变更的级联影响范围

### 4.2 HCM 中的典型图查询

**继任链路：**
```
"张三能继任谁？通过什么发展路径？中间需要补什么能力？"

(张三)-[:具备能力]->(能力A)
(目标岗位)-[:要求能力]->(能力B)
路径 = 能力A 到 能力B 的最短发展路径
```

**人才寻访：**
```
"找一个做过销售管理、懂 AI、带过 50 人团队、在华东区有经验的人"

MATCH (p:Employee)-[:曾任]->(j:Job {type:'销售管理'})
WHERE (p)-[:具备]->(s:Skill {name:'AI'})
  AND (p)-[:管理过]->(t:Team {size:'>50'})
  AND (p)-[:工作过]->(r:Region {name:'华东'})
RETURN p
```

**组织影响分析：**
```
"如果把技术中心拆成两个部门，影响哪些人？"

从 技术中心 节点出发，遍历所有关联关系：
  → 直属员工（汇报线变更）
  → 虚线汇报员工（矩阵关系变更）
  → 继任计划（继任关系可能失效）
  → 绩效关系（考核归属变更）
  → 预算归属（成本中心变更）
```

**能力差距分析：**
```
"这个岗位要求 10 项能力，团队 30 个人各自差什么？谁最接近？"

岗位要求能力集合 R = {能力1, 能力2, ..., 能力10}
每个员工能力集合 E_i = {能力x, 能力y, ...}
差距 = R - E_i
匹配度 = |R ∩ E_i| / |R|
```

### 4.3 图数据库的局限

- 事务能力弱（不适合做主库）
- 写入性能低于关系型
- 运维复杂度高
- 团队学习成本（Cypher 语法）
- 生态不如 PostgreSQL 成熟

## 5. 推荐架构：关系型主库 + 图分析层

### 5.1 架构图

```
┌──────────────────────────────────────────────────┐
│                  应用层 / AI 层                     │
│  ┌────────────────┐  ┌─────────────────────────┐  │
│  │ 业务 CRUD 操作  │  │ AI 分析 & 关系推理       │  │
│  │ (事务型操作)    │  │ (分析型查询)             │  │
│  └───────┬────────┘  └────────────┬────────────┘  │
│          │                        │               │
│  ┌───────┴────────┐  ┌────────────┴────────────┐  │
│  │  PostgreSQL    │  │  Neo4j / NebulaGraph    │  │
│  │  主库（写+读）  │  │  分析层（只读）          │  │
│  │                │  │                         │  │
│  │  • 员工主数据   │  │  • 组织关系网络          │  │
│  │  • 薪酬考勤    │  │  • 能力/技能图谱         │  │
│  │  • 审批流程    │  │  • 继任关系网络          │  │
│  │  • 标签资产    │  │  • 协作关系              │  │
│  │  • 规则任务    │  │  • 职业发展路径          │  │
│  │  • 事务数据    │  │  • 影响传播分析          │  │
│  └───────┬────────┘  └─────────────────────────┘  │
│          │                        ↑               │
│          └──── CDC / 定时同步 ─────┘               │
└──────────────────────────────────────────────────┘
```

### 5.2 数据分工原则

| 放关系型数据库 | 放图数据库 |
|--------------|-----------|
| 员工主数据（姓名、工号、入职日期等） | 组织关系网络（含矩阵汇报、虚线汇报） |
| 薪酬、考勤、社保 | 能力/技能图谱（依赖、前置、相似） |
| 标签资产、规则、任务 | 继任关系网络（继任链、发展路径） |
| 审批流程、操作日志 | 人才流动路径（调动、轮岗历史） |
| 绩效评分原始数据 | 协作关系（项目协作、跨部门合作） |
| 所有需要 ACID 事务的写操作 | 跨维度关系查询和路径分析 |

### 5.3 数据同步策略

```
PostgreSQL (主库)
    │
    ├── CDC (Change Data Capture)
    │   └── 实时捕获变更 → 增量同步到图数据库
    │       适用：员工调动、组织变更等高频变更
    │
    └── 定时批量同步
        └── 每日全量/增量同步
            适用：能力评估结果、绩效数据等低频变更
```

同步原则：
- 图数据库只读，所有写操作通过关系型主库
- 同步延迟容忍度：分析场景可接受分钟级延迟
- 同步失败不影响主库业务

### 5.4 图数据库节点和关系设计

**节点类型：**

```
(:Employee)      员工
(:Organization)  组织
(:Position)      岗位
(:Skill)         技能/能力
(:Certification) 认证
(:Project)       项目
(:Tag)           标签
(:Training)      培训课程
```

**关系类型：**

```
(:Employee)-[:BELONGS_TO]->(:Organization)        所属组织
(:Employee)-[:REPORTS_TO]->(:Employee)             直线汇报
(:Employee)-[:DOTTED_LINE_TO]->(:Employee)         虚线汇报
(:Employee)-[:MENTORS]->(:Employee)                导师关系
(:Employee)-[:SUCCEEDS]->(:Position)               继任关系
(:Employee)-[:HAS_SKILL {level, years}]->(:Skill)  具备技能
(:Employee)-[:PARTICIPATED_IN]->(:Project)         参与项目
(:Employee)-[:HAS_TAG]->(:Tag)                     拥有标签
(:Position)-[:REQUIRES {level}]->(:Skill)          岗位要求
(:Skill)-[:DEPENDS_ON]->(:Skill)                   技能依赖
(:Skill)-[:SIMILAR_TO {score}]->(:Skill)           技能相似
(:Training)-[:DEVELOPS]->(:Skill)                  培训发展
(:Organization)-[:PARENT_OF]->(:Organization)      组织层级
```

## 6. AI 集成方案

### 6.1 双数据源工具设计

AI 工具分两类，根据问题类型自动选择数据源：

| 工具类型 | 数据源 | 典型问题 |
|---------|--------|---------|
| SQL 查询工具 | PostgreSQL | "张三的薪资是多少""本月有多少人请假" |
| 图查询工具 | Neo4j | "张三能继任哪些岗位""技术中心拆分影响谁" |
| 混合查询工具 | 两者 | "找一个 P8 以上、会 AI、带过大团队的人"（SQL 筛基础条件 + 图查关系） |

### 6.2 System Prompt 设计

```
你是 HCM 系统的 AI 助手，可以访问两个数据源：

1. 结构化数据库（PostgreSQL）：
   - 存储员工主数据、薪酬、考勤、标签、规则等事务数据
   - 适合精确查询、统计聚合、CRUD 操作
   - 使用 SQL 查询工具

2. 关系图谱（Neo4j）：
   - 存储组织关系、能力图谱、继任网络、协作关系
   - 适合路径分析、影响评估、相似度匹配、多跳关系查询
   - 使用图查询工具

选择数据源的原则：
- 查具体数据（薪资、考勤、标签）→ SQL
- 分析关系（继任、影响、路径）→ 图
- 复合查询（条件筛选 + 关系分析）→ 先 SQL 筛选，再图分析
```

### 6.3 AI 查询路由示例

| 用户问题 | 路由 | 原因 |
|---------|------|------|
| "张三的基本信息" | SQL | 单表查询 |
| "技术中心有多少人" | SQL | 简单聚合 |
| "张三能继任谁" | 图 | 多跳路径遍历 |
| "如果李四离职，影响哪些继任计划" | 图 | 影响传播分析 |
| "找一个 P7 以上、会 Python、在销售部待过的人" | 混合 | SQL 筛 P7 + 图查技能和经历 |
| "技术中心和销售中心之间有哪些人员交集" | 图 | 跨组织关系发现 |
| "张三和李四之间有什么关联" | 图 | 动态路径发现 |
| "本月薪资发放总额" | SQL | 纯结构化计算 |

## 7. 实施路径

### 7.1 阶段规划

```
阶段一（当前）：纯关系型
  └── PostgreSQL 承载所有数据
  └── 打标系统、基础 HCM 功能
  └── AI 通过 SQL 工具查询

阶段二（扩展期）：引入图分析层
  └── 新增 Neo4j / NebulaGraph
  └── 同步组织关系、能力数据到图
  └── AI 新增图查询工具
  └── 实现继任分析、能力差距分析

阶段三（成熟期）：图谱深化
  └── 协作关系、项目关系入图
  └── 技能相似度自动计算
  └── 职业发展路径推荐
  └── 组织变更影响模拟
```

### 7.2 引入图数据库的时机判断

满足以下任一条件时考虑引入：

- 出现 4 层以上 JOIN 的查询需求，且数据量超过 10 万
- 业务需要"路径推荐"类功能（继任路径、发展路径）
- 业务需要"影响分析"类功能（组织调整影响、人员离职影响）
- 业务需要"动态关系发现"（两个对象之间的未知关联）
- 现有 SQL 查询响应时间超过 2 秒且无法通过索引优化

### 7.3 技术选型建议

| 图数据库 | 适用场景 | 说明 |
|---------|---------|------|
| Neo4j | 中小规模（百万节点以内） | 生态最成熟，Cypher 语法友好，社区版免费 |
| NebulaGraph | 大规模（亿级节点） | 国产，分布式架构，性能好 |
| Apache AGE | 轻量集成 | PostgreSQL 扩展，不需要额外部署 |

**对于 HCM 系统，推荐 Apache AGE 作为起步方案** — 它是 PostgreSQL 的图扩展，不需要额外部署图数据库，不需要数据同步，直接在现有 PostgreSQL 上启用图查询能力。等规模和需求增长后再迁移到独立的 Neo4j。

## 8. 总结

| 阶段 | 数据架构 | 适用范围 |
|------|---------|---------|
| 打标系统 | 纯 PostgreSQL | 标签、规则、任务、结果 |
| 基础 HCM | PostgreSQL + Apache AGE | 加入组织关系图、能力图谱 |
| 完整 HCM | PostgreSQL + Neo4j/NebulaGraph | 继任、协作、路径推荐、影响分析 |

核心原则：**关系型做事务主库，图做分析层。写操作走关系型，关系分析走图。AI 根据问题类型自动路由数据源。**
