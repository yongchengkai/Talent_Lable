# 人才打标系统 - PRD 与原型交叉审查报告

> 基于 PRD 重构版全文与 11 个原型页面代码的逐项交叉比对，识别 PRD 描述不清、缺失或与代码实现不一致的问题。

---

## 一、"AI 原生产品"的定位没有说清楚

**现状：** PRD 1.1 提到"AI 与结构化规则共用同一套标签资产"，但通篇来看 AI 只是一个"打标方式"——和结构化规则并列的另一种规则类型。这不是 AI 原生，这是"传统系统 + AI 插件"。

**具体缺失：**

### 1.1 规划智能体（planning-agent）在 PRD 里完全没有提及

代码里已有 993 行的 `planning-agent.js`，能根据业务场景自动生成标签类目、标签定义和打标规则的完整方案，还能调用模型 API (`/api/planning-agent/analyze`)。但 PRD 对这个能力只字未提——不在 MVP 范围里，也不在后续迭代里。

### 1.2 AI 对话生成 DSL 的能力描述不足

代码里 `rules.html` 的编辑弹窗右侧已实现 AI 对话区，能从自然语言生成规则 DSL。PRD 5.2 只用一句话带过。缺少以下说明：

- AI 生成的 DSL 草稿需不需要人工确认才能保存？
- AI 对话的上下文管理策略是什么？
- AI 生成的规则和人工编写的规则在治理上有没有区别？

### 1.3 AI 语义规则的技术约束未定义

代码里 `smart-tags.html` 的实现是：配置输入字段 + 写语义规则描述 + 从描述中提取候选标签。PRD 没有回答：

- AI 模型是外部调用还是内置？
- 语义规则的 prompt 模板谁来维护？
- AI 打标的批量执行策略是什么（逐条调用 vs 批量推理）？

### 1.4 建议

如果系统定位为"AI 原生"，PRD 应增加独立章节说清楚 AI 在系统中的角色层次：

- 规划层：自动生成标签方案（planning-agent）
- 规则层：AI 辅助编写规则（AI 对话生成 DSL）
- 执行层：AI 语义打标（语义规则执行）
- 质量层：AI 结果校验（置信度、人工纠错）

## 二、"什么字段不用打标"的规范散落在两个文档里

**现状：** `标签生成方式说明.md` 把判断标准讲得很清楚（不打标的三类：原始事实字段、纯实时计算字段、低价值重复字段），但 PRD 里几乎没有引用这套规范。

**具体问题：**

- PRD 1.4 的"标签生成方式总览"只有一个三行表格，没有把"不打标"的判断标准写进去
- 如果只看 PRD，规则管理员不知道"职级"该不该打成标签
- `标签生成方式说明.md` 和 PRD 之间没有明确的引用关系，容易脱节
- 代码里 `planning-agent.js` 的 `buildPlan()` 函数硬编码了标签分类逻辑（通过关键词匹配 `flags.management`、`flags.tech`、`flags.campus`），和文档里的判断框架没有对齐

**建议：** PRD 应在 2.3（打标规则）之前增加"标签设计原则"章节，把"什么不打、什么打、怎么打"的决策树直接嵌入 PRD。

## 三、规则 DSL 的语法规范完全没有定义

**现状：** 代码里已实现一套 DSL 语法（`rules.html` 的 `parseDslDraft` 函数），但 PRD 里对规则表达能力的描述只有四行。

**代码中已实现的 DSL 语法：**

```
rule "规则名"
when
  条件表达式 -> tag "标签名"
  AND 条件表达式 -> tag "标签名"
then
  mode store
end

explain:
  自然语言解释
```

支持的能力：
- `rule "规则名"` 声明
- `when` / `then` / `end` 块结构
- `AND` / `OR` / `NOT` 逻辑连接
- `-> tag "标签名"` 输出语法
- `mode store` 存储指令
- `explain:` 解释块

**PRD 中的描述（2.4 节）仅有：**
- 单值匹配
- 枚举匹配
- 区间映射
- 多分支映射

**缺失：** DSL 语法定义、保留字、运算符列表、错误处理规则、导入导出格式标准。

---

## 四、正式打标和模拟打标的差异描述不够精确

**现状：** PRD A.5 和 A.6 分别描述了模拟打标和正式打标，但代码里 `recalc-tasks.html` 基本是 `rule-publish.html` 的复制品（连 mock 数据都一样），只做了文案替换。

**PRD 约束 vs 代码实现的矛盾：**

- PRD 说"正式打标只允许选择已发布状态规则"，但代码里正式打标的规则选择列表包含了草稿和已停用的规则
- PRD 提到"待提交→已提交"两步确认流程，代码里没有实现提交按钮和状态流转

**PRD 缺失的说明：**

- 正式打标的"待提交→已提交"两步确认流程的具体交互
- 正式打标结果和模拟打标结果在数据层面的隔离方式
- 重算任务和正式打标任务的关系（同一入口还是不同入口？）

---

## 五、冲突消解机制在 PRD 里缺失

**现状：** 数据库草案定义了 `final_decision` 字段（HIT / REJECTED / CONFLICT_DROPPED），并提到"精度 > 优先级 > 发布时间"的冲突消解策略。但 PRD 里完全没有提到冲突消解。

**具体问题：**

- 同一个人被多条规则命中同一类目下的不同标签时怎么处理？
- PRD 3.2 明确说"类目只负责分类，不承担互斥裁决"，那互斥裁决在哪里做？
- 规则优先级字段在 PRD 的规则对象定义里没有出现
- 数据库草案的冲突消解策略没有在 PRD 业务流程中体现

## 建议修复优先级

| 优先级 | 问题 | 原因 |
|--------|------|------|
| P0 | 冲突消解机制补入 PRD | 核心业务逻辑空白，直接影响打标结果正确性 |
